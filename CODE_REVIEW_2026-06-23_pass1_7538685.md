# NearMe Code Review — pass 1

**Commit:** `7538685`
**Reviewer:** Claude Code
**Date:** 2026-06-23

## Summary

The NearMe extension scaffold is well-structured for a MW 1.39 special page: Cargo NEAR queries are delegated cleanly to a service class, the API module validates inputs properly, and the JS frontend correctly escapes all user-controlled output via `mw.util.escapeHtml()`. There are four actionable bugs — three PHP and one JS — plus a handful of suggestions and nits. The most impactful bug is in the JS (`mw.config.get` wrong key prefix), which causes the admin-configured radius and limit to be silently ignored in production.

## Bugs (must fix)

### B1 — PHP: Implicit nullable parameter deprecation (PHP 8.1)
**File:** `includes/ApiCargoNearby.php:26`

```php
// Current (triggers E_DEPRECATED on PHP 8.1)
public function __construct( ApiMain $main, string $action, NearbyQueryService $queryService = null ) {

// Fix
public function __construct( ApiMain $main, string $action, ?NearbyQueryService $queryService = null ) {
```

PHP 8.1 deprecated implicit nullable parameters. On the Saintapedia target (PHP 8.1+) this emits a deprecation notice on every API call.

### B2 — PHP: `dieWithError` substitutes `$1` incorrectly for `nearme-error-unknown-table`
**File:** `includes/ApiCargoNearby.php:61` and `:72`

```php
// Current — $data arg, not i18n params; $1 never substituted
$this->dieWithError( 'nearme-error-unknown-table', 'unknown-table', [ 'table' => $tableFilter ] );

// Fix — wrap as array: [messageKey, param1, param2, ...]
$this->dieWithError( [ 'nearme-error-unknown-table', $tableFilter ], 'unknown-table' );
```

`ApiBase::dieWithError( $msg, $code, $data )` — the third argument is API error `$data`, not i18n message parameters. As written, the error reads **"Unknown Cargo table: $1"** in the API response instead of **"Unknown Cargo table: Saints"**. Same issue applies to both call sites.

### B3 — JS: `mw.config.get` called with wrong key prefix
**File:** `modules/nearby-api.js:60-61`

```js
// Current — keys with 'wg' prefix are not found; fallbacks always used
var radius = options.radius || mw.config.get( 'wgNearMeDefaultRadius', 10000 );
var limit  = options.limit  || mw.config.get( 'wgNearMeDefaultLimit', 50 );

// Fix — ResourceLoader module config keys use the exact names from extension.json
var radius = options.radius || mw.config.get( 'NearMeDefaultRadius', 10000 );
var limit  = options.limit  || mw.config.get( 'NearMeDefaultLimit', 50 );
```

The `wg` prefix applies to core MediaWiki config variables loaded by the startup module. Extension ResourceModule `config` keys are exposed with their exact name as listed in `extension.json`. Since the key `wgNearMeDefaultRadius` doesn't exist, `mw.config.get` returns the hardcoded fallback every time — silently ignoring whatever the wiki admin set in `LocalSettings.php`.

---

### B4 — PHP: Non-numeric `gscoord` parts silently cast to `0.0`
**File:** `includes/ApiCargoNearby.php:44-47`

```php
// Current — (float)'' === 0.0, which passes range check and queries near lat=0,lon=0
$lat = (float)$parts[0];
$lon = (float)$parts[1];
if ( $lat < -90 || $lat > 90 || $lon < -180 || $lon > 180 ) { ... }

// Fix — add numeric check before casting
if ( !is_numeric( $parts[0] ) || !is_numeric( $parts[1] ) ) {
    $this->dieWithError( [ 'apierror-badparameter', 'gscoord' ], 'bad-coord' );
}
$lat = (float)$parts[0];
$lon = (float)$parts[1];
```

A request like `gscoord=|5.0` produces `$parts[0] = ''`, which casts to `0.0`. That passes the range check and silently queries near lat=0, lon=0 (Gulf of Guinea). The server-side range check alone is not enough — add `is_numeric()` first.

## Suggestions (should fix)

### S1 — PHP: Exception handling too narrow in `queryAll`
**File:** `includes/NearbyQueryService.php:138`

```php
// Current — only catches MWException
} catch ( MWException $e ) {

// Fix — also catch DB errors and general exceptions
} catch ( \Exception $e ) {
```

`CargoSQLQuery::run()` can throw `Wikimedia\Rdbms\DBError` (e.g. table doesn't exist in the DB yet, Cargo not installed for that table). `DBError` does not extend `MWException`, so it bubbles up as an uncaught exception and produces a 500 for all Cargo tables if any one fails.

---

### S2 — PHP: `wfLogWarning()` deprecated since MW 1.35
**File:** `includes/NearbyQueryService.php:139`

```php
// Current
wfLogWarning( 'NearMe: Cargo query failed for ' . $source['table'] . ': ' . $e->getMessage() );

// Fix
wfDebugLog( 'NearMe', 'Cargo query failed for ' . $source['table'] . ': ' . $e->getMessage() );
```

`wfLogWarning()` triggers `E_USER_WARNING` (visible in PHP error logs). `wfDebugLog()` routes to MW's logging system (configurable per-channel) and is the standard pattern in MW 1.39+.

---

### S3 — JS: `new mw.Title()` throws on invalid titles in `toCard()`
**File:** `modules/nearby-api.js:39`

```js
// Current — throws if row.title is not a valid MW title
var title = new mw.Title( row.title );

// Fix — use newFromText which returns null on failure
var title = mw.Title.newFromText( row.title );
if ( !title ) { return null; }
// ... and filter nulls after map:
// rows.map( toCard ).filter( Boolean )
```

If a single Cargo row has a page title that `mw.Title` considers invalid, the entire `.map(toCard)` call throws, `getPagesAtCoordinates` rejects, and the error handler shows a generic "Could not load" message instead of the partial results from other rows.

---

### S4 — JS: `showButtonDisabled` never reset after permission-denied
**File:** `modules/ext.NearMe.js:125-128`

Once the user denies geolocation, `this.showButtonDisabled = true` is set and the button is permanently disabled for the page session. If the user then grants browser permission and reloads, `showButtonDisabled` starts as `false` again — so reloading does fix it. But a note/link ("Enable location in browser settings, then reload") in the permission-denied message would help. Separately, consider resetting `showButtonDisabled` to `false` before each attempt in `showNearby()` so recoverable errors (timeout, position-unavailable) don't prevent retries.

---

### S5 — JS: Unverified behavior of `mw.router.addRoute` exit callback
**File:** `modules/ext.NearMe.js:143-154`

```js
mw.router.addRoute(
    coordinateRegex,
    function ( lat, lon ) { /* enter */ },
    function () { /* exit — clear results */ }   // ← third arg
);
```

MW 1.39's `mediawiki.router` wraps a lightweight router whose `addRoute` accepts `(path, callback)`. A third "on-exit" callback is not documented and is likely silently ignored. If so, navigating from `#/coord/40.44,-79.99` back to `#` leaves stale results on screen. Verify against MW 1.39 source or test manually.

## Nits (optional)

### N1 — `nearme-error-guidance` message loaded but never rendered
**File:** `extension.json:59`, `i18n/en.json`

`nearme-error-guidance` is listed in the module's `messages` array and defined in `en.json`, but no JS code calls `mw.msg('nearme-error-guidance')`. Either render it below the error `<div>` alongside `nearme-error`, or remove it from both files to avoid translator confusion.

### N2 — README states PHP >= 7.4 but Saintapedia targets 8.1
**File:** `README.md:18`

Update the requirements table to `>= 8.1` to match the actual deployment target (CLAUDE.md). PHP 7.4 reached EOL in December 2022.

### N3 — `jquery` not listed as explicit module dependency
**File:** `extension.json:44-48`

`location-provider.js` uses `$.Deferred()`. jQuery is available transitively via `mediawiki.api` in MW 1.39, but listing `jquery` explicitly in `dependencies` is best practice and guards against future dependency pruning.
